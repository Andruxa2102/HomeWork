## ETL-пайплайн сбора данных из API в Data Vault 2

пайплайн полного цикла: от первичной загрузки сырых данных из API до построения версионированного хранилища в концепции Data Vault 2.0 с отслеживанием истории изменений по каждой организации.



## Архитектура проекта

my_nsi_project/
├── pyproject.toml
├── README.md
├── pytest.ini                                  # файл конфигурации pytest
├── .env.example                                # Шаблон для переменных пайплайна
├── docker-compose.yml
├── src/
│   ├── db.py                                   # Функции для работы с PostgreSQL
│   ├── loader.py                               # Парсер + загрузка в БД
│   └── core_builder.py                         # Оркестратор расчета слоев Data Vault 2
├── sql/
│   ├── init.sql                                # Запросы на создание объектов БД
│   └── core/
│       ├── load_hub_organization.sql           # Запрос для расчета nsi.Hub_Organization
│       ├── load_sat_organization_attrs.sql     # Запрос для расчета nsi.Sat_Organization_Attrs
│       └── load_sat_organization_changes.sql   # Запрос для расчета nsi.Sat_Organization_Changes
└── test/
    ├── conftest.py                             # Инструменты и pytest
    ├── test_api_units.py                       # тест API-клиента
    ├── test_loader.py                          # общий тест loader.py
    └── test_version_logic.py                   # тест логики проверки версии

Вопросы и предложения по развитию архитектуры пайплайна:

1. Добавить run_id (идентификатор пайплайна загрузки): Ввести в таблицы колонку run_id, как ссылку на nsi.log_pipeline_runs.run_id. Это обеспечит сквозное отслеживание происхождения данных (Data Lineage) и позволит изолированно перепроверять результаты конкретных запусков.

2. Добавить load_id (идентификатор времени загрузки): Использовать системную колонку load_at из nsi.raw_medical_organizations в качестве load_id (идентификатора бизнес-события загрузки), что позволит отслеживать состояния данных по времени без жесткой зависимости от последовательных суррогатных ключей:
- на случай перезагрузки данных;
- для повышения производительности запросов исторической аналитики;
